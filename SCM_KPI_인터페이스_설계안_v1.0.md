## 1. 협력사 재고 품절 KPI

### 데이터 소스
- **베이스**: MRP (`applObFUJy5o025oQ`)
- **주 테이블**: `sync_parts`
- **보조 테이블**: `sales_status_history`, `stock_answer_history`

### 필요 신규 필드 (sync_parts 테이블에 추가)

| 필드명 | 타입 | 수식 / 설명 |
|--------|------|------------|
| `품절시작일` | Lookup | `sales_status_history` 연결 → 가장 최근 `판매상태 = 품절`인 `변경일` |
| `품절일수` | Formula | `IF({판매상태}='품절', DATETIME_DIFF(TODAY(), {품절시작일}, 'days'), 0)` |
| `대체협력사_여부` | Formula | `IF(COUNTA({partner}) > 1, "있음", "없음")` |
| `재입고예정일` | Lookup | `stock_answer_history` → 최신 `협력사 재입고 일정` |

> ⚠️ `품절시작일`은 현재 sales_status_history에서 직접 lookup이 어려울 수 있음.  
> 대안: `판매상태 변경일` 필드를 sync_parts에 lastModifiedTime으로 추가하거나,  
> 기존 `fldmyQMUEF4ilUjN3(판매상태 변경일)` 필드 활용 가능.

### 인터페이스 구성 — "협력사 재고 품절 현황"

**타입**: Grid View (인터페이스 탭 내 Report / Grid element)

**필터 조건**:
- `판매상태` = 품절 **OR** `파츠품절 Status` = 품절

**표시 필드 순서**:

| 순서 | 필드 | 비고 |
|------|------|------|
| 1 | 파츠명 | 기본키 |
| 2 | 협력사 (partner lookup) | 현재 담당 협력사 |
| 3 | 판매상태 | 품절 강조 (빨간색 조건부) |
| 4 | 파츠품절 Status | 세부 품절 상태 |
| 5 | 품절시작일 | 위 신규 필드 |
| 6 | 품절일수 | 위 신규 필드. 30일 이상 경고색 |
| 7 | 대체협력사_여부 | 위 신규 필드 |
| 8 | 재입고예정일 | 협력사_재입고예정일자 |
| 9 | 협력사_재입고예정수량 | 기존 필드 |
| 10 | 재고담당자 | 담당자 표시 |

**그룹화**: `재고담당자` 기준 (담당자별 품절 현황 파악)

**정렬**: `품절일수` DESC (오래된 품절 먼저)

**KPI 요약 지표** (인터페이스 상단 Summary 블록):
- 현재 품절 파츠 수: `COUNT(판매상태 = 품절)`
- 30일 이상 장기 품절: `COUNT(품절일수 >= 30)`
- 대체 협력사 없는 품절 건: `COUNT(대체협력사_여부 = 없음)`
- 재입고 미확정 건: `COUNT(재입고예정일 = empty)`

| `fldVhPSugTYwKVNwa` | 수주처 | 협력사명 |
| `fldiEfna4G3t41J9L` | 표준적요명 | 파츠코드+파츠명 포함 |
| `fldggjY1oIsQlwm01` | 발주지시수량 | 매입 수량 |

| `fldl2IxTwmFHnnicZ` | 매입결산진행 | 결제 월 정보 |

### 필요 신규 필드 (task 테이블에 추가)

| 필드명 | 타입 | 수식 |
|--------|------|------|
| `매입총금액` | Formula | `{발주지시수량} * {공급가액 단가}` |
| `발주년도` | Formula | `YEAR({과업지시일자})` |
| `발주월` | Formula | `MONTH({과업지시일자})` |
| `발주주차` | Formula | `WEEKNUM({과업지시일자})` |
| `파츠코드` | Formula | `LEFT({표준적요명}, 9)` (PNA로 시작하는 5자리 숫자 추출) |
| `파츠명_표시` | Formula | `표준적요명에서 코드 제거한 제품명` |

> ⚠️ `표준적요명` 형식: `PNA51440_데일리짐색(단품)_A1` 구조  
> 파츠코드 = `LEFT(표준적요명, 9)`, 파츠명 = `MID(표준적요명, 11, ...)` 으로 추출 가능

### 인터페이스 구성 — "제품별 판매 KPI"

#### View 1: 주간 매입 현황 (Weekly Purchase)

**타입**: Grid View  
**그룹화**: `발주주차` + `발주년도`  
**집계**: `매입총금액` SUM, `발주지시수량` SUM  
**필터**: 현재 연도 (`발주년도 = 2026`)  
**정렬**: 과업지시일자 DESC

**표시 필드**:

| 필드 | 설명 |
|------|------|
| 표준적요명 (파츠명_표시) | 제품명 |
| 수주처 | 협력사 |
| 발주지시수량 | 주간 매입 수량 |
| 공급가액 단가 | 단가 |
| 매입총금액 | 발주지시수량 × 단가 |
| 과업지시일자 | 발주일 |
| 매입결산진행 | 결제 상태 |

#### View 2: 제품별 매입 집계 (Top 판매 제품)

**타입**: Grid View  
**그룹화**: `파츠코드` (또는 `표준적요명`)  
**집계 필드**: `매입총금액` SUM, `발주지시수량` SUM  
**정렬**: `매입총금액` DESC  
**필터**: 과업지시일자 = 최근 4주 (또는 선택 기간)

**KPI 요약 지표** (인터페이스 상단):
- 주간 총 매입금액
- 주간 발주 건수
- TOP 5 매입 제품 (매입금액 기준)
- 전주 대비 매입 증감률 (formula)

#### View 3: 원가율 계산용 뷰 (매입단가 기준)

**타입**: Grid View  
**그룹화**: `파츠코드`  
**필드**: 파츠코드, 파츠명_표시, 공급가액 단가 평균, 발주지시수량 합계, 매입총금액 합계  
**정렬**: 매입총금액 DESC

## 3. 원가율 분석

### 데이터 소스 구조

| 항목 | 소스 | 비고 |
|------|------|------|
| 매입단가 (공급가) | SERPA task.공급가액 단가 | 협력사 발주 단가 |
| 판매단가 (공급가) | 코스트 베이스 (`app2fbwlQiRFUoMV5`) | 고객 공급단가 |
| 표준원가 | SCM sync_parts의 `100개_표준원가` 등 | 수량별 원가 기준 |
| 매입금액 | SERPA task.매입총금액 (신규) | 기간별 집계 |

> **원가율 = 공급가액 단가(매입) / 판매단가(공급가)** 로 정의

### 원가율 분석 구현 전략

두 베이스 간 직접 연결이 에어테이블에서 어렵기 때문에 아래 2단계 접근 추천:

**Step 1**: SERPA task 뷰에서 `파츠코드` 기준으로 평균 공급가액 단가 집계  
**Step 2**: 코스트 베이스에서 해당 파츠코드의 판매단가 확인  
**Step 3**: 원가율 = 매입단가 / 판매단가 (수동 비교 또는 외부 시트 연동)

### 인터페이스 구성 — "원가율 분석"

#### View 1: 원가율 위험 품목

**타입**: Grid View (SERPA task 테이블 기반)  
**필드**: 파츠코드, 파츠명_표시, 공급가액 단가, 발주수량, 매입총금액  
**정렬**: 매입총금액 DESC  
**조건부 색상**:
- 원가율 > 70% → 빨간색 (위험)
- 원가율 50~70% → 노란색 (주의)
- 원가율 < 50% → 초록색 (양호)

> ⚠️ 원가율 계산을 에어테이블 내에서 자동화하려면 SERPA task 테이블에 판매단가 lookup 필드를 추가하거나, 코스트 베이스와 연동하는 Automation / Zapier 필요.  
> 단기적으로는 `공급가액 단가` 기준 내림차순 정렬로 고비용 제품 우선 검토.

#### View 2: 고매출-저마진 제품 모니터링

**목적**: 매입금액은 크지만 마진율이 낮은 제품 식별  
**구현**: 매입총금액 TOP 30 추출 → 코스트 베이스에서 판매단가 대조

**추가 추천 필드 (SERPA task에 추가)**:
- `판매단가_참조` (Number): 코스트 베이스에서 수동 입력 또는 automation으로 동기화
- `원가율` (Formula): `{공급가액 단가} / {판매단가_참조}`
- `원가율_등급` (Formula): 
  ```
  IF({원가율} > 0.7, "🔴위험", 
    IF({원가율} > 0.5, "🟡주의", "🟢양호"))
  ```

#### KPI 요약 지표
- 원가율 위험 품목 수 (>70%)
- 전체 평균 원가율
- 매입금액 TOP 5 품목별 원가율
- 전월 대비 원가율 변동 상위 품목

## 4. 협력사 관리 탭 추가수정

### 4-1. 25년/26년 월별 협력사 지표 추이 비교

#### 데이터 소스
- **베이스**: SERPA (`appkRWtF2j99XgBTq`)
- **테이블**: `지출결의` (`tblKOBouWywz9XlyU`)
- **주요 필드**: `수주처(협력사)`, `결제일`, `총금액_합계`, `지출현황`

#### 확인된 데이터 구조 (지출결의 테이블)

| 예시 레코드 | 수주처 | 결제일 | 총금액 |
|------------|--------|--------|--------|
| 26.3.9-3.13_그리닉 | 그리닉 | 2026-03-12 | 164,630 |
| 24.10.7-10.11_(주)승화파트너스 | (주)승화파트너스 | 2024-10-10 | 31,854,680 |

#### 신규 필드 추가 (지출결의 테이블)

| 필드명 | 타입 | 수식 |
|--------|------|------|
| `결제년도` | Formula | `YEAR({결제일})` |
| `결제월` | Formula | `MONTH({결제일})` |
| `결제년월` | Formula | `YEAR({결제일}) & "-" & LPAD(MONTH({결제일}), 2, "0")` |

#### 인터페이스 구성 — "협력사별 연도 매입액 추이"

**타입**: Grid View + Chart Element (에어테이블 인터페이스)

**Grid View 설정**:
- 그룹화 1: `수주처` (협력사별)
- 그룹화 2: `결제년월`
- 집계: `총금액_합계` SUM
- 필터: `결제년도` IN (2025, 2026)

**차트 설정** (에어테이블 Chart element):
- 타입: 꺾은선 그래프 (Line Chart)
- X축: 결제년월
- Y축: 총금액_합계 합산
- 색상 구분: 수주처(협력사)별
- 연도 필터: 25년 vs 26년 비교

**추가 KPI 지표**:
- 협력사별 25년 총 매입액 vs 26년 YTD 매입액
- 전년 동기간 대비 증감률
- 협력사별 매입 비중 (파이차트)

### 4-2. 하도급법 90일 결제대금 미지급 현황

#### 데이터 소스
- **베이스**: SERPA (`appkRWtF2j99XgBTq`)
- **테이블**: `task` (`tbllJFVBoFjmbQXLN`)
- **기산점 필드**: `과업지시일자` (확인 완료)
- **결제완료 판단 필드**: `매입결산진행` (결제 월 정보 있으면 완료로 판단)

#### 신규 필드 추가 (task 테이블)

| 필드명 | 타입 | 수식 / 설명 |
|--------|------|------------|
| `90일_기한일` | Formula | `DATEADD({과업지시일자}, 90, 'days')` |
| `90일초과여부` | Formula | `IF(AND(IS_BEFORE({90일_기한일}, TODAY()), {매입결산진행}=''), "⚠️ 초과", "")` |
| `D-day` | Formula | `DATETIME_DIFF({90일_기한일}, TODAY(), 'days')` |
| `발주담당자확인` | 담당자(People) | 담당자별 추가 확인용 신규 필드 |
| `90일확인메모` | 멀티라인 텍스트 | 지연 사유 및 대응 내용 기록 |

#### 인터페이스 구성 — "하도급법 90일 미지급 현황"

**필터 조건**:
- `매입결산진행` = 비어있음 (결제 미완료)  
- AND `과업지시일자` = 90일 이전 (`{과업지시일자} < DATEADD(TODAY(), -90, 'days')`)

또는 위험 임박 포함:
- `D-day` < 14 (14일 이내 기한 도래 포함)

**표시 필드**:

| 순서 | 필드 | 비고 |
|------|------|------|
| 1 | task_id | 발주 ID |
| 2 | 수주처 | 협력사명 |
| 3 | 표준적요명 | 품목명 |
| 4 | 발주지시수량 | 수량 |
| 5 | 매입총금액 | 금액 |
| 6 | 과업지시일자 | 발주 생성일 (기산점) |
| 7 | 90일_기한일 | 법적 기한 |
| 8 | D-day | 잔여일 (마이너스 = 초과) |
| 9 | 90일초과여부 | ⚠️ 표시 |
| 10 | 발주담당자확인 | 담당자별 확인 |
| 11 | 90일확인메모 | 지연 사유 메모 |

**정렬**: `D-day` ASC (가장 급한 건 상단)

**그룹화**: `발주담당자확인` (담당자별 확인 현황)

**KPI 요약**:
- 90일 초과 건수
- 90일 초과 미지급 총액
- 14일 이내 기한 도래 건수 (위험 임박)

### 4-3. 품질이슈 제품별/협력사별 집계 및 리뷰

#### 데이터 소스 현황
- SCM 베이스 `22 하반기_ 협력사 데이터` 테이블: 품질이슈 건수, 수량이슈 건수, 재제작 건수 (22년 데이터)
- QC WIKI 베이스 (`appcXrs6d9ZiGB93j`): 품질 이슈 데이터 (현재 접근 권한 확인 필요)

> ⚠️ **확인 필요**: 현재 품질이슈를 어느 베이스/테이블에서 기록하고 있는지 파악 후 설계 확정 필요.  
> QC WIKI 베이스 구조 파악 후 아래 설계 완성 예정.

#### 잠정 설계안 (QC 데이터 소스 확인 후 수정)

**이상적인 품질이슈 집계 구조**:

```
품질이슈 레코드
  ├── 발생일자
  ├── 협력사명 (sync_partners 연결)
  ├── 파츠명 (sync_parts 연결)
  ├── 이슈 유형 (품질불량/수량오류/납기지연/기타)
  ├── 불량 수량
  ├── 원인 분류 (작업자 오류/설계 문제/자재 불량/공정 문제)
  ├── 대응 내용 (재제작/반품/폐기/크레딧)
  ├── 재제작 금액
  └── 처리 완료 여부
```

**인터페이스 구성**:
- View 1: 협력사별 품질이슈 집계 (협력사 그룹, 이슈건수/재제작금액 합계)
- View 2: 파츠별 품질이슈 집계 (파츠 그룹, 불량률 계산)
- View 3: 원인별 분석 (이슈 유형/원인 분류 기준)
